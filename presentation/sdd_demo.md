Spec Coding -> adds some SDLC componennts/aspects


SDD (Spec Driven development)

Prompt spec (JIRA) -> Requirement -> decision( happy then edit else design) - design -> happy -> implememnt and edit

As from speckit

specify(what and why and not how only Functional requirements) -> clarify(optional- this helps to change any requirements generated by ask qna) -> plan (add the technical details like what code changes and table level or ui flag) -> tasks(it breaks down the plan into small tasks) -> implement 


Different from other dev cycles

Traditional : Code -> Documenttaion
TDD  : Test -> Code
SDD : Specs -> Code





The Speckit Flow (What Each Step Does)
| Step            | Command               | Purpose                              | What YOU Provide                                                                                   | What LLM Generates                                                                             |
| --------------- | --------------------- | ------------------------------------ | -------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| 1. Constitution | /speckit.constitution | Set project-wide technical rules     | Your stack (Spring Boot 3.5.x, Java 21, MapStruct, SOLID, observability, API standards) devyoutube | constitution.md with immutable technical guardrails github.github+1                            |
| 2. Specify      | /speckit.specify      | Capture functional requirements      | Plain user story: "What business problem to solve + why" github.github+1youtube                    | spec.md with user stories, acceptance criteria, functional constraints (tech-agnostic) b-lab+1 |
| 3. Clarify      | /speckit.clarify      | Resolve ambiguities in spec          | Answers to LLM's questions about gaps github.githubyoutubedocs.runmaestro                          | Adds ## Clarifications section to spec with your answers baked in docs.runmaestro+1            |
| 4. Plan         | /speckit.plan         | Create technical implementation plan | Tech stack direction, architecture choices, constraints github.github+1                            | Detailed plan.md with file structure, data models, API contracts, technology selection x-cmd+1 |
| 5. Tasks        | /speckit.tasks        | Break plan into executable tasks     | Usually nothing (just run it) dev                                                                  | tasks.md with dependency ordering, exact file paths, TDD structure dev                         |
| 6. Implement    | (normal chat)         | Write code                           | Review tasks, then implement                                                                       | Code changes                                                                                   |




Quick Reference: What Goes Where

| Information Type                                  | Where to Put It                          |
| ------------------------------------------------- | ---------------------------------------- |
| Business problem, user story, acceptance criteria | /speckit.specify github.github+1         |
| Ambiguities, edge cases, missing details          | /speckit.clarify answers github.github+1 |
| Spring Boot 3.5.x, Java 21, MapStruct, SLF4J      | Constitution (project-wide) dev+1        |
| Table changes, code changes, file paths           | Plan (feature-specific) github.github+1  |
| DTOs, mappers, repository patterns                | Plan → LLM generates x-cmd+1             |
| Task breakdown, TDD structure                     | Tasks → LLM generates dev                |
